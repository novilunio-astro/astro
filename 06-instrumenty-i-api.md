# Инструменты, библиотеки и API для астро-продукта

Техническая часть базы: чем считать карты, транзиты и матрицу программно, и на какие готовые сервисы смотреть.

## 1. Ядро расчётов: Swiss Ephemeris

**Swiss Ephemeris** (Astrodienst) — индустриальный стандарт расчёта положений планет (точность на тысячелетия, основан на эфемеридах NASA JPL). Практически все серьёзные астро-приложения считают на нём.

- Документация: https://www.astro.com/swisseph/swisseph.htm
- Лицензия двойная: **AGPL** (бесплатно, если открываешь код) или **коммерческая лицензия** Astrodienst (для закрытого кода — платная, разовая). Для стартапа это ключевое лицензионное решение.
- Экосистема обёрток: https://github.com/topics/swiss-ephemeris

| Язык | Библиотека |
|------|-----------|
| Python | `pyswisseph` — https://pypi.org/project/pyswisseph/ |
| JS/Node | `swisseph` (порт, https://github.com/arturania/swisseph), `sweph` |
| C/C++ | оригинальная библиотека Astrodienst |

Поверх pyswisseph есть высокоуровневые open-source библиотеки (расчёт целых карт, аспектов, домов): **Kerykeion** (Python, генерирует карты + SVG), **Immanuel** (Python), **flatlib** (традиционная астрология).

**Альтернатива без лицензионных вопросов:** Moshier ephemeris (чуть ниже точность, public domain) — используется в части JS-портов; для MVP точности хватает.

| RoxyAPI | Расчёты на Swiss Ephemeris через REST | Платный/фримиум — https://roxyapi.com/blogs/swiss-ephemeris-explained-developers |
| Vedika (гайд «как построить свой astrology API») | Разбор архитектуры такого API | Статья — https://vedika.io/blog/how-to-build-astrology-api |

Стратегически: свой расчётный сервис на pyswisseph/Kerykeion дешевле на масштабе и не создаёт зависимость от вендора; сторонний API быстрее для MVP.

## 4. Матрица судьбы: расчёт

Внешние API не нужны — вся математика в несколько строк кода (приведение даты к арканам 1–22, см. `04-matrica-sudby.md`). Ценность продукта — база трактовок и визуализация октаграммы (SVG).

## 5. Технические грабли (проверено индустрией)

1. **Часовые пояса и историческое летнее время** — главный источник неверных карт. Использовать IANA tzdata + для старых дат проверять исторические зоны (в pyswisseph время подаётся в UT — конверсию делать самим).
2. **Геокодинг места рождения** — нужен справочник городов с координатами (GeoNames — бесплатный).
3. **Системы домов** — давать минимум Placidus + Whole Sign; в высоких широтах Placidus ломается (за полярным кругом) — нужен fallback.
4. **Ретроградность и тройные проходы транзитов** — при генерации календаря событий учитывать 3 точных аспекта, а не один.
5. **Лицензия Swiss Ephemeris** (AGPL vs коммерческая) — решить до релиза закрытого кода.
6. **Форматы данных**: хранить момент рождения как (локальное время + tz + координаты), а не готовый UTC — чтобы можно было пересчитывать при исправлении данных.

## 6. Идеи для изучения конкурентов

Co–Star, The Pattern, Chani, Sanctuary (EN-рынок); Astro.com как «профессиональный» полюс; в RU-сегменте — сервисы из обзора vc.ru выше. Смотреть: онбординг (сбор даты/времени/места), структуру бесплатного контента, пуш-механики на транзитах, модель подписки.
